<objective>
Run a Feynman-style active recall loop: generate a challenge, collect the user's attempt, assess mastery, teach gaps, and iterate until the user demonstrates understanding (Level 3+) or pauses.

**Orchestrator role:** Parse the concept and options, manage the mastery-bounded loop, spawn `gpd-tutor` for challenges, `gpd-mastery-assessor` for evaluation, and `gpd-explainer` for gap teaching. Delegate all state management (session, memory, FSRS, Bjork, prereqs) to the `gpd-learning` MCP server.

**Why subagent:** Each role (challenge generation, assessment, explanation) requires fresh context and verification independence. The tutor must not see the assessment criteria. The assessor must not be biased by the tutor's expectations.

**Why MCP:** State management via `gpd-learning` provides FSRS-6 spaced repetition scheduling, Bjork dual-strength memory tracking, and schema migration — testable, runtime-portable, and separate from research state.
</objective>

<process>

## 0. Validate Context

```bash
CONTEXT=$(gpd --raw validate command-context learn "$ARGUMENTS")
if [ $? -ne 0 ]; then
  echo "$CONTEXT"
  exit 1
fi
```

## 1. Parse Request

Extract the target concept and options from `$ARGUMENTS`:
- Concept: the physics topic to learn
- `--type`: recall, derive, or apply (default: derive)
- `--review`: show prior sessions for this concept instead of starting a new one

## 2. Check Prior Sessions

If prior sessions exist for this concept, show the last mastery level and session summary.

## 3. Execute the Learn Workflow

Follow the learn workflow from `@{GPD_INSTALL_DIR}/workflows/learn.md` end-to-end.

## 4. Return Results

Show the mastery journey, final level, gaps closed, and suggested next steps.
</process>

<success_criteria>

- [ ] Concept parsed and validated
- [ ] For new concepts: user offered choice between full explanation first or direct challenge
- [ ] Prior sessions checked and displayed if they exist
- [ ] Challenge generated by `gpd-tutor` and attempt collected
- [ ] Attempt assessed by `gpd-mastery-assessor` with mastery level assigned
- [ ] If below mastery (Level 3): adaptive policy applied (difficulty/type adjustment) before re-attempt
- [ ] If below mastery (Level 3): gaps taught by `gpd-explainer`, re-attempt offered
- [ ] Loop terminates on mastery (Level 3+) or user pause
- [ ] Session state persisted via `gpd-learning` MCP (SESSION.json + MEMORY.json with FSRS + Bjork)
- [ ] FSRS card initialized on mastery (Level 3+), spaced review scheduled
- [ ] Learning log updated with session summary via `end_session`
- [ ] User receives mastery journey, next steps, and review schedule
</success_criteria>
